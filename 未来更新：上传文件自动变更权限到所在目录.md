# 需求

我为https://github.com/Eugeny/tabby 软件开发了一款扩展插件，其中有个功能是重构了sftp，我已经完成了功能，现在遇到一个问题：如果我是root用户登陆，当我上传文件到服务器后，上软的文件也变成了root。我希望上传的文件夹及文件夹都跟随所在文件夹的权限自动变更

# 参考解决方案：

# 一、你真正想实现的效果

你实际上想要的是：

```
上传到哪个目录
→ 文件 owner/group 自动与该目录一致
```

例如：

```
/var/www/html
owner: www-data:www-data
```

root 上传：

```
test.php
```

希望：

```
www-data:www-data
```

而不是：

```
root:root
```

------



# 二、推荐实现方案（行业标准）

流程：

```
1. root 上传临时文件
2. 查询目标目录 owner/group
3. 自动 chown
4. 自动 chmod（可选）
```

例如：

```
目标目录:
/var/www/html

stat:
uid=33
gid=33
```

上传后：

```
chown 33:33 test.php
```

------

# 三、核心技术实现

你需要：

## 第一步：获取目标目录 owner/group

SFTP 可获取 stat 信息。

Node.js 常见库：

- ssh2
- ssh2-sftp-client

都支持：

```
sftp.stat(path)
```

返回：

```
{
  uid: 33,
  gid: 33,
  mode: 16877
}
```

------

## 第二步：上传完成后 chown

OpenSSH SFTP 协议支持：

```
SETSTAT
FSETSTAT
```

里面可以设置：

- uid
- gid
- mode

如果你的 SFTP 重构没实现这个功能，问题可能就在这里。

------

# 四、很多 SFTP 库的坑

很多 Node SFTP 库：

- 只实现 chmod
- 没完整实现 chown
- 或服务端不支持
- 或 Windows 下被忽略

特别：

```
sftp.chown()
```

很多库根本没封装。

但 ssh2 原生协议层其实支持。

------

# 五、真正的大坑：OpenSSH 的限制

即使客户端发送：

```
SETSTAT uid/gid
```

OpenSSH 也未必允许。

因为：

# 只有 root 才能 chown

普通用户：

```
Operation not permitted
```

但你现在正好是 root。

所以：

------

# 六、大概率失败的原因

你很可能：

## 1. 上传时机不对

文件还没 close：

```
upload stream 未结束
→ chown 失败
```

------

## 2. chown 后又被覆盖

某些上传逻辑：

```
tmp file
rename
```

rename 后 owner 又变化。

------

## 3. 目录递归时遗漏

上传文件夹：

```
mkdir
upload
```

新建目录本身已经 root:root。

后续文件即使修复：

目录 owner 还是错。

------

## 4. 使用 chmod 代替 chown

很多人误以为：

```
chmod
```

能改 owner。

实际上：

```
chmod = 权限
chown = 所属用户
```

完全不同。

------

# 七、正确的完整实现

建议：

# 上传文件

```
1. stat 父目录
2. 上传文件
3. close handle
4. chown(uid,gid)
5. chmod(继承 mode 可选)
```

------

# 上传目录

```
1. mkdir
2. chown 目录
3. chmod
4. 递归上传
```

------

------

# 八、最佳实践（你插件应该这样做）

我建议你在 Tabby 插件里增加：

# “继承目录权限”

选项：

```
[x] 上传后自动继承目标目录 owner/group
[x] 继承 mode
```

实现：

```
stat(parent)
→ upload
→ chown
→ chmod
```

这是最稳定的。

# 九、可能遇到的特殊问题

## SELinux

如果：

```
getenforce
```

是：

```
Enforcing
```

那么：

```
chown 后
context 仍然不对
```

网站仍可能 403。

需要：

```
restorecon
```

或者：

```
chcon
```

------

# 十、最终结论

你的需求：

# 技术上完全可实现

但核心点是：

```
不能依赖 Linux 自动继承 owner
```

必须：

# 上传后主动 chown

真正稳定实现：

```
stat(parent)
→ upload
→ close
→ chown(uid,gid)
→ chmod(mode)
```

# 十一、后续注意

1. 需要注意断点续传功能，
2. 断点后权限继承功能是否还生效
3. 如果当前账户不是root，但是需要变更超过当前账户的权限，需要给出提示框输入密码后执行。
